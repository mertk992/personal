01

Why This Exists

Large capital programs like satellite constellations generate hundreds of spend requests per quarter. Each one needs budget validation, financial impact analysis, policy checks, historical benchmarking, and routing to the right approver. Today that workflow is spreadsheets, email chains, and manual lookups across half a dozen systems.

This prototype demonstrates how an agentic AI system could compress that process. Instead of a fixed pipeline, the agent receives a request and a toolbox, then decides what data it needs, gathers it, and produces a recommendation with full reasoning transparency.

Demo context: This prototype uses Blue Origin's publicly announced TeraWave satellite constellation as a scenario. All financial figures, contracts, vendor names, and program data are entirely synthetic. Nothing here reflects actual Blue Origin financials or internal processes. Built to demonstrate the architecture, not to model reality.

02

What Makes It Agentic

"Agentic" is a specific claim. Here's what it means in this system and how it differs from a standard chatbot or a hardcoded pipeline.

Hardcoded Pipeline This System
Step order Fixed: step 1 → 2 → 3 → 4, every time Agent decides which tools to call and in what order
Adaptability Same path regardless of input A $5M routine request triggers 3 tool calls; a $50M request on a tight budget triggers 7
Follow-up None — runs once and outputs Agent reads tool results and chains additional calls based on findings
Reasoning Business logic in code Agent explains why it called each tool and how it interpreted results
API mechanism Prompt → response Claude tool_use API: prompt → tool call → execute → result → next call → … → final answer

The core loop: Claude gets the request + 6 tool definitions. It decides what it needs to know, calls a tool, reads the result, then decides what to do next. Repeat until it has enough information to produce a recommendation. The reasoning path changes per query.

04

How the Agent Loop Works

This is the core of the system. Every analysis follows the same loop, but the path through it is different each time because the agent decides.

1

Request arrives

A CapEx request or natural language query enters the system. The agent receives the request along with definitions for all 6 tools it can call.

2

Agent plans

Claude reads the request, reasons about what data it needs, and selects a tool to call. For example: "This is a $25M Ground Segment request — let me check the budget pool first."

3

Tool executes

The system runs the selected tool (e.g., queries budget status) and returns the result to Claude as structured data.

4

Agent interprets and chains

Claude reads the tool result and decides what to do next. If the budget is tight, it might call the variance tool to check for underspend elsewhere. If it's a vendor-related request, it searches program documents for contract context. This chaining is not scripted — it emerges from the agent's reasoning.

5

Repeat until done

Steps 2–4 repeat. A routine request might take 3 tool calls. A complex one might take 6 or 7. The agent stops when it has enough information.

6

Recommendation

Claude synthesizes all findings into a recommendation (GO / CONDITIONAL / DEFER / NO-GO) with specific rationale, conditions, and risk flags. The full reasoning chain is visible in the UI so the analyst can audit every step.

05

Agent Toolbox

Six tools defined in Claude's tool_use API format. Each tool has a name, description, and JSON input schema. The agent chooses which to call based on the request.

check_budget
Queries budget pool status: total, spent, committed, available, and utilization percentage.
SAP S/4HANA — GL • Cost Centers
run_financial_impact
Estimates NPV, ROI, payback period, progress score, and risk retirement score for a proposed spend.
Anaplan — Forecast Models • BvA
search_documents
RAG search over program documents: contracts, vendor memos, capital policies, technical reports.
SharePoint — Contracts • Policies
get_comparable_requests
Finds historical CapEx requests with similar characteristics for precedent and benchmarking.
Coupa — PO History • Spend Data
check_approval_routing
Determines approval tier, required approver, and SLA based on amount and urgency classification.
ServiceNow — Approval Workflows
get_variance_status
Checks YTD budget vs. actual for a workstream, surfaces anomalies and trend data.
Snowflake — Data Warehouse • FP&A

In this prototype, each tool calls a synthetic data layer. In production, the tool implementations would be thin API wrappers over the enterprise systems listed above. The agent code stays the same — only the data source changes.

06

Beyond CapEx

The CapEx workflow is the flagship demo, but the same agentic engine powers the full platform. Other capabilities built on the same architecture:

Monte Carlo Simulation
Correlated uncertainty modeling using Cholesky decomposition across capex, revenue, and timeline variables. P10/P50/P90 distributions.
FP&A Variance Dashboard
Monthly budget vs. actual with automated anomaly detection, contextual commentary, and drill-down by workstream.
Document Intelligence (RAG)
TF-IDF vector search over synthetic contracts, vendor memos, and policies. AI answers grounded in source documents with citations.
DCF Modeling
Full 12-year cash flow projection with NPV, IRR, payback period, terminal value, and a progress-per-dollar capital efficiency framework.
